# v1.2.0 features — deleteclip, presets, replay input suppression, spectator-quality freecam

**Status:** design approved 2026-07-07. Targets Iustitia **v1.2.0** — the first public release after the public v1.1.0. (The prior internal `build.gradle.kts` bumps to 1.2.1 / 1.2.2 during development were never publicly shipped; this release consolidates the whole batch — solid-world playclip, lazy chunk baking, and the four features below — as the single public v1.2.0. So `build.gradle.kts` is set to `1.2.0`, a downgrade from the internal 1.2.2 placeholder.)
**Author:** brainstormed with the user; spec by Claude.

This spec covers four features the user requested after shipping 1.2.2:

1. `/ius deleteclip <name>` (alias `/ius delclip`) — delete a saved clip by name.
2. `/ius preset <name>` — apply a named config preset. Five built-ins (Strict / Standard / Lenient / Debug / Moderation) plus user-created custom presets that persist like built-ins.
3. During any active replay or playclip, the local player's gameplay inputs must not reach the live server (full spectator-like: no move / look / break / place / use / interact / attack / swing). Chat, commands, and inventory stay usable.
4. The playclip FREECAM must feel like real spectator mode — fly freely with WASD + mouse, no collision, no black-out inside blocks, no floating hand. Reimplemented to match the Zergatul FreeCam v26.2 reference the user supplied.

## Standing constraints (carried from prior work, unchanged except as noted)

- Client-sided Fabric anticheat (MC 1.21.11, yarn 1.21.11+build.6, Java 21, Kotlin/JVM, Fabric Loom 1.17.11, fabric-language-kotlin 1.13.9+kotlin.2.3.10, YACL 3.8.2).
- Read-only / fail-open / **no outgoing packets** / no enforcement / no world edits.
  - **SCOPED EXCEPTION (user-authorized for v1.2.0):** suppressing the local player's own outgoing *gameplay* packets during an active replay/playclip is explicitly authorized (feature #3). This is suppression of the local observer's own inputs only — never other players, never detection, never any packet outside the active-replay window. Chat / commands / inventory / hotbar-slot packets are intentionally NOT suppressed so the mod stays usable while spectating.
- New display/UX config fields use `if (o.has(...))` guards — **no `CONFIG_VERSION` bump** for new fields.
- The clip *file* format version is separate from `CONFIG_VERSION` (unchanged at 4; clip format stays at 6).
- All mixin bodies `try { … } catch (_: Throwable) {}`.
- Mixin static fields `private`.
- Mixin config `defaultRequire: 1` (a target mismatch fails launch).
- Chunk-unloaded-never-FP.
- Mixin class names use underscores (`iustitia_*`).
- FREECAM is only available for a chunk-bearing playclip (`ReplayState.chunks != null`).
- Runtime-only-verifiable mixin/render targets: the build can't confirm a mixin applies or a render path works — only a live client run can. Every such target is gated so a failure degrades (live world shows through / slower / camera stays on player) rather than crashing.

## Architecture decisions (approved)

### A. Freecam rework → pure camera-override (match the Zergatul FreeCam v26.2 reference)

Drop the entity-based `FreeCamera` + `setCameraEntity` entirely. Store six pose primitives on `ReplayState`; write them straight onto the vanilla `Camera` each frame; redirect mouse-look to the pose; drive movement by free-space integration (noclip, no gravity — spectator feel); force the camera's render-state `isSpectator()` true so looking inside/behind blocks doesn't black out; hide the hand.

Why not keep the entity port: the user reported the current FREECAM "isn't actually a freecam where the user can move around freely" and "is very laggy." The entity approach has more runtime failure modes (world-tick, camera-follow interpolation, entity init) than the camera-override approach, and the reference mod proves the camera-override approach delivers the spectator feel the user wants with less code.

### B. Input suppression → input-freeze + mouse-look redirect + packet-cancel mixin (hybrid)

Three pieces, all gated on `ReplayState.active` (every cam mode — FREE / FOLLOW / POV / FREECAM):

1. **Input-freeze** — extend the existing `ClientPlayerEntityMixin` so `input.playerInput = PlayerInput.DEFAULT` at `tickMovement` HEAD runs during *any* active replay, not just FREECAM. The player never walks (client- or server-side) while a replay/clip runs.
2. **Mouse-look** — for FREECAM, `FreecamEntityMixin` already redirects `Entity.changeLookDirection` to the freecam pose (re-pointed from the removed entity to the new pose primitives). For FREE / FOLLOW / POV, the local player's client yaw may still change (FOLLOW orbit math reads it), but the packet-cancel below stops it reaching the server.
3. **Packet-cancel mixin** — a new mixin on the outgoing-packet chokepoint cancels `PlayerMoveC2SPacket`, `PlayerActionC2SPacket`, `PlayerInteractBlockC2SPacket`, `PlayerInteractItemC2SPacket`, `PlayerInteractEntityC2SPacket`, `HandSwingC2SPacket`, `PlayerInputC2SPacket` while `ReplayState.active`. This is the piece the reference mod does NOT have — it covers break / place / use / interact / attack / swing, which the reference's input-swap alone doesn't suppress. Chat, commands, inventory clicks, hotbar slot changes, and abilities packets are NOT cancelled (true spectator can still chat / open inventory / pick hotbar).

Why hybrid (not pure packet-cancel): without the input freeze the player would animate walking client-side (jittery) and generate-then-cancel packets every tick. The freeze is cleaner and matches the reference. The packet-cancel is the belt-and-suspenders for movement + the sole mechanism for the non-movement gameplay actions.

### C. Preset storage → built-in code templates + custom JSON preset files

Built-ins are code-defined `IustitiaConfig` templates (a fresh `IustitiaConfig()` + per-preset overrides — full snapshots, so apply is a uniform field copy). Custom presets are full config snapshots serialized to `%APPDATA%/.iustitia/presets/<name>.json` (or game-dir `.iustitia/presets`) via `/ius createpreset <name>`. `/ius preset <name>` resolves built-in first, then custom, applies by copying fields into the live config in place, saves, reloads. Custom presets persist across restarts like built-ins. `/ius deletepreset <name>` removes a custom one (built-ins cannot be deleted). `/ius presets` lists all.

## Section 2 — Preset system

### `config/PresetManager.kt` (new)

`object PresetManager`:

- `builtInNames: List<String> = listOf("strict","standard","lenient","debug","moderation")`.
- `builtIn(name): IustitiaConfig?` — returns a fresh `IustitiaConfig()` with the per-preset overrides applied (see table below). Returns null for unknown names.
- `apply(name): Boolean` — resolve built-in → else load custom JSON. On hit, copy every preset-content field into `ConfigManager.config` **in place** (mutate the live config object, never replace the reference — other subsystems hold it). Then `ConfigManager.save()` + `Iustitia.onConfigReloaded()`. Returns false when the name is neither built-in nor a custom file.
- `saveCustom(name): Boolean` — serialize the current `ConfigManager.config` to `presets/<name>.json` using the same Gson `JsonObject` shape `ConfigManager.toJson` already produces (reuse that serializer so a custom preset round-trips byte-for-byte with the live config). Refuse to overwrite a built-in name. Sanitize `name` with the same `Regex("[^A-Za-z0-9_.-]")` → `_` that `ClipStore.pathFor` uses.
- `deleteCustom(name): Boolean` — `Files.deleteIfExists` on the custom path; refuse + return false for built-in names.
- `listCustom(): List<String>` — filenames in the presets dir, extension stripped, sorted.
- `listAll(): List<String>` — built-ins first, then customs.
- Custom-preset dir: `FabricLoader.getInstance().gameDir.resolve(".iustitia/presets")` (mirrors `ClipStore`'s `.iustitia/clips`).

### Apply semantics — what is and isn't preset content

A preset captures **detection calibration + display/UX + replay/sonar** — everything that defines how the mod behaves. It deliberately excludes per-user / runtime state:

**Copied (preset content):** `enabled`, `verbose`, `alertLevel`, `alertBatching`, `alertBatchWindowTicks`, `audioCues`, `audioVolume`, `audioNuclear`, `lagSuppressAlerts`, `nametagBurstPulse`, `compactMode`, `evidenceWindowTicks`, `transcriptPanel`, `lagHudIcon`, `confidenceHud`, `targetHighlight`, `ghostTrail`, `watchFollowCam`, `burstSparks`, `hoverTooltip`, `tabListBadge`, `nametagPrefixes`, `nametagGreenEnabled`, `alertsEnabled`, `alertThrottleTicks`, `joinGraceTicks`, `legitScaffoldStrictGates`, `replayCapture`, `replayHideLive`, `replayPlayerModels`, `replayRelocate`, `clipTerrain`, `clipChunkWorld`, `clipChunkRadius`, `clipChunkRenderDistance`, `sonarAlerts`, `sonarVolume`, and all 36 `CheckConfig` slices (`enabled`/`setbackVL`/`decay`/`threshold`).

**Excluded (per-user / runtime):** `configVersion` (always the code default — never let a preset downgrade calibration migration), `mutedChecks`, `mutedPlayers`, `wizardCompleted`, `persistenceEnabled` (a moderator's persistence preference is environmental, not part of a detection profile).

Apply copies field-by-field into the live config object (not `config = template` — that would orphan the reference other subsystems hold). For each `CheckConfig` slice, copy `enabled`/`setbackVL`/`decay`/`threshold` into the existing slice object in place (same pattern the live `toggle`/`threshold` commands already use at `IustitiaCommand.kt:328,346`).

### Built-in preset definitions

Detection baseline for every built-in = a fresh `IustitiaConfig()` (the tuned 36-check defaults). Per-preset overrides:

| Preset | Checks | setbackVL scale | Chat detail | Visuals | sonar | replay | nametags | Other |
|---|---|---|---|---|---|---|---|---|
| **strict** | all 36 ON (defaults) | ×0.5 from defaults (alert sooner) | `verbose=true`, `alertLevel=2`, `alertBatching=false`, `compactMode=false` | all OFF | off | on | on (`nametagPrefixes=true`, `nametagGreenEnabled=true`) | — |
| **standard** | all 36 ON (defaults) | defaults (×1.0) | `alertLevel=1`, `alertBatching=true`, `compactMode=false`, `verbose=false` | all OFF | off | on | on | — |
| **lenient** | blatant ON; subtle/corroboration-tier OFF (see list) | ×2.0 from defaults (only sustained blatant alerts) | `alertLevel=1`, `alertBatching=true`, `compactMode=false`, `verbose=false` | all OFF | off | on | on | — |
| **debug** | all 36 ON (defaults) | defaults (×1.0) | `verbose=true`, `alertLevel=2`, `alertBatching=true`, `compactMode=false` | all ON | on | on | on | `audioCues=true`, `audioNuclear=true`, `persistenceEnabled` NOT touched (excluded), `transcriptPanel=true` |
| **moderation** | all 36 ON (standard defaults) | defaults (×1.0) | `verbose=false`, `compactMode=true`, `alertLevel=1`, `alertBatching=true` | all OFF **except** `hoverTooltip=true`; `confidenceHud=false` | off | on | on | — |

"All visuals OFF" = `targetHighlight=ghostTrail=watchFollowCam=burstSparks=hoverTooltip=tabListBadge=nametagBurstPulse=lagHudIcon=confidenceHud=transcriptPanel=false`.
"All visuals ON" (debug) = all of the above `true`.

**Lenient disabled-check list (subtle / corroboration-tier):** `hitsWithoutSwing`, `hitFlick`, `triggerbot`, `packetGap`, `rotationSnapBack`, `wTap`, `keepSprint`, `jumpOnHurt`, `aimWrap`, `scaffoldRotation`. These are the signals most prone to false-positive on legitimate-but-aggressive play; the blatant-tier checks (reach, killAura, criticals, maceSmash, fly/speed envelopes, noFall, etc.) stay on. This list is a judgment call and is called out in the user-facing docs as tunable.

**setbackVL scaling:** scaled from the **default** values (a fresh `IustitiaConfig()`), not the user's current tuning — so a preset always means the same thing regardless of prior tweaks. `×0.5` = `default.setbackVL * 0.5`; `×2.0` = `default.setbackVL * 2.0`. Custom presets capture the user's exact tuned values, so `/ius createpreset` is the way to save a hand-tuned setup.

### No `CONFIG_VERSION` bump

## Section 3 — Input suppression (full spectator-like, all replay modes)

### Modify `mixin/ClientPlayerEntityMixin.kt`

Widen the gate from `cameraMode == FREECAM` to `ReplayState.active` (any mode). The body is unchanged (`input?.playerInput = PlayerInput.DEFAULT`). This stops the player walking during `/ius replay` and `/ius playclip` in every cam mode, not just FREECAM.

KDoc update: the mixin now suppresses movement during *any* active replay/clip, so the player never moves server-side while spectating a rewind/clip. The FREECAM-only wording is dropped.

### New `mixin/ClientConnectionMixin.kt` (packet cancel — the one genuinely new mixin beyond the reference)

`@Mixin(ClientConnection::class)` (yarn `net.minecraft.network.ClientConnection`), `@Inject(method = ["send"], at = [At("HEAD")], cancellable = true)`. When `ReplayState.active`, cancel the packet if it's an instance of any of:

- `net.minecraft.network.packet.c2s.play.PlayerMoveC2SPacket` (look + position — covers `Full`, `LookAndOnGround`, `PositionAndOnGround` subclasses)
- `net.minecraft.network.packet.c2s.play.PlayerActionC2SPacket` (break / place start-cancel / drop / swap)
- `net.minecraft.network.packet.c2s.play.PlayerInteractBlockC2SPacket`
- `net.minecraft.network.packet.c2s.play.PlayerInteractItemC2SPacket`
- `net.minecraft.network.packet.c2s.play.PlayerInteractEntityC2SPacket` (attack / interact)
- `net.minecraft.network.packet.c2s.play.HandSwingC2SPacket`
- `net.minecraft.network.packet.c2s.play.PlayerInputC2SPacket` (the 1.21+ movement-input packet — belt-and-suspenders alongside the input freeze)

Allow everything else through (chat `ChatMessageC2SPacket`, command `ChatCommandC2SPacket`, inventory `ClickSlotC2SPacket`/`ButtonClickC2SPacket`/`PickItemC2SPacket`, hotbar `PickFromInventoryC2SPacket`/slot-change, `ClientStatusC2SPacket`, `ClientSettingsC2SPacket`, `UpdateSelectedSlotC2SPacket`, abilities, etc.). The filter is a closed allow-by-type list of the gameplay packets above; anything not in the list passes.

**Target note (runtime-only-verifiable):** the exact chokepoint is `ClientConnection.send(Packet<?>)` in 1.21.11 yarn. There may be an overload `send(Packet, PacketCallbacks)` / a future-flavor variant. The mixin lists all `send` overloads that take a `Packet` as the first param; the inject handler inspects the first arg via `callbackInfo`-shimmed args. If the build/runtime rejects the target, the fallback is to mixin `ClientPlayNetworkHandler.sendPacket` instead (the network handler's send dispatcher). Both are validated at build time against the named jar; `defaultRequire: 1` means a mismatch fails launch rather than silently not applying. The body is try/caught so a runtime throw leaves the packet going through (fail-open = the player can act during a replay, never a crash).

**Add to `iustitia.mixins.json`** `client` array: `ClientConnectionMixin`.

### Restore on stop

`ReplayState.stop()` already restores perspective + drops the freecam pose. The input mixin self-disables when `active` goes false (one volatile read per tick); the packet-cancel mixin likewise. No saved input state to restore — we never replaced `input`, only zeroed a field `Input.tick()` rebuilds next tick; we never sent a suppressed packet. The player's client yaw may have drifted from the server's last-known during the replay, but MC rotation is client-authoritative — the next real rotation packet after stop syncs it with no rubberband. Position never drifted (input frozen + move packets cancelled).

### Scope safety

The packet-cancel reads only `ReplayState.active` (a `@Volatile Boolean`) and the packet's runtime type. It acts only on the local observer's own outgoing packets during the active-replay window. It does not touch incoming S2C, other players, detection, or any packet outside the window. This is the user-authorized scoped exception to the no-outgoing-packets rule.

## Section 4 — Spectator-quality freecam (rework, pure camera-override)

### Replace entity-based `FreeCamera.kt` with pose primitives on `ReplayState`

`ReplayState` changes:

- Remove `freeCamera: FreeCamera?`, `FREECAM_ENTITY_ID`, `enterFreecam()`/`exitFreecam()` (entity-spawn form), and the `FreeCamera`-tick path in `tickFreecam()`.
- Add six pose primitives (client-thread-written, render-thread-read — `@Volatile`):
  - `@Volatile var fcX: Double = 0.0`
  - `@Volatile var fcY: Double = 0.0`
  - `@Volatile var fcZ: Double = 0.0`
  - `@Volatile var fcYaw: Float = 0f`
  - `@Volatile var fcPitch: Float = 0f`
  - `@Volatile var freecamActive: Boolean = false` (private set; distinct from `cameraMode == FREECAM` so the render thread can read it without re-deriving)
- `enterFreecam()` (private, called from `setCameraMode` on FREECAM transition): seed `fcX/fcY/fcZ` from the local player's pos and `fcYaw/fcPitch` from the player's look, plus a 2-block back-off along the look vector (matches the reference's "back off so you don't start inside your own face"). Set `freecamActive = true`. Idempotent.
- `exitFreecam()` (private): `freecamActive = false`; pose primitives left at last value (harmless; re-enter reseeds). Idempotent.
- `tickFreecam()` (called from `Iustitia.onClientTick`): no-op unless `freecamActive && active`. Reads the held vanilla `KeyBinding`s (`mc.options.forwardKey/backKey/leftKey/rightKey/jumpKey/sneakKey.isPressed`, `sprintKey.isPressed`) — public API, no mixin — and advances the pose by free-space integration: camera-relative WASD along `fcYaw`/`fcPitch` (same proven math as the old `FreeCamera.doMotion`), jump=+Y, sneak=−Y, sprint×1.2, no collision (noclip fly), no gravity. Speed ~0.5 blocks/tick walking, 1.2 sprinting. Tiny dead-zone to avoid diagonal 1.41×. Fail-open try/caught.

`setCameraMode` keeps the FREECAM-enter/leave branching but calls the new `enterFreecam()`/`exitFreecam()`. `stop()` calls `exitFreecam()`.

### `mixin/CameraMixin.kt` — FREECAM branch writes the pose

The FREECAM branch in `tryReplay` (currently a no-op that relies on vanilla following the `FreeCamera` entity) becomes an active pose write — the same `setPos`/`setRotation` the POV/FOLLOW branches already use:

```
ReplayState.CameraMode.FREECAM -> {
    if (!ReplayState.freecamActive) return false
    setRotation(ReplayState.fcYaw, ReplayState.fcPitch)
    setPos(Vec3d(ReplayState.fcX, ReplayState.fcY, ReplayState.fcZ))
}
```

**Coordinate space (decided):** the freecam pose primitives are stored in **relocated/screen space** directly — `enterFreecam` seeds from the player's live pos (already screen space) and `tickFreecam` advances in screen space. The camera write uses the pose verbatim with NO `relocOffset` shift. This matches the reference (freecam flies in the world you see) and keeps the freecam intuitive (the user flies in screen space, not recorded space). The chunk world + ghosts are rendered with the shared `relocOffset` translate (`screen = recorded - camPos + relocOffset`), so a screen-space freecam pos lines up with them. POV/FOLLOW still shift the *recorded* focus snap by `relocOffset` because those snaps are in recorded space; FREECAM doesn't, because its pose is already in the space the camera writes.

### `mixin/FreecamEntityMixin.kt` — redirect to the pose (not the entity)

The `@Inject` on `Entity.changeLookDirection` keeps its gate (`active && cameraMode == FREECAM && this === mc.player`) but, instead of forwarding to `freeCamera.changeLookDirection(...)`, it updates the freecam pose:

```
ReplayState.fcYaw = ...   // apply yawDelta to fcYaw (MC yaw convention)
ReplayState.fcPitch = ... // apply pitchDelta to fcPitch, clamp ±90
ci.cancel()
```

The exact yaw/pitch update math mirrors vanilla `Entity.changeLookDirection`'s body (so the freecam look feels identical to vanilla mouse-look): yaw += yawDelta * 0.15-ish sensitivity factor, pitch += pitchDelta * factor, pitch clamped to ±90. The reference applies the same deltas vanilla would. Re-entrancy: the mixin only fires on `this === mc.player`; the pose update doesn't call back into `changeLookDirection`, so no recursion. `@Suppress("CAST_NEVER_SUCCEEDS")` on `this as Entity` stays.

### Spectator render-state — no black-out inside blocks

The reference forces the camera's render state `isSpectator()` true so the camera doesn't black out when it's inside/behind solid blocks (vanilla culls the block the camera is in unless spectator). Two options, decided at build time against the 1.21.11 named jar:

- **Preferred:** extend `CameraMixin` (or add a tiny companion mixin on the camera render-state path) to make the camera report spectator while `freecamActive`. The reference does this in `MixinCamera` on the render-state extraction.
- **Fallback:** if the render-state hook is fragile on 1.21.11, skip it for v1.2.0 and document "camera may black out inside blocks" as a known limitation — the freecam still flies, just don't nose into walls. The chunk world is fullbright + flat-lit so it reads clearly outside walls either way.

This is runtime-only-verifiable. The preferred path is attempted first; the fallback is a one-line pivot if the build/runtime rejects the target.

### Hide the hand (spectator has no hand)

The reference hides the first-person hand in freecam via a `GameRenderer.renderItemInHand` redirect. Iustitia's `EntityRendererMixin` already hides the local player model in FREECAM. Extending hand-hide is a small addition to the existing render mixin or a tiny new `GameRendererMixin` redirect on the hand-render call. Runtime-only-verifiable; if the hand-render hook is fragile, hiding the hand is a documented follow-up (the freecam works without it — you'd just see a floating hand, cosmetic only).

### Files removed / added for the freecam rework

- **Remove:** `replay/FreeCamera.kt` (the entity port). Drop its imports in `ReplayState.kt`.
- **Modify:** `replay/ReplayState.kt` (pose primitives + enter/exit/tick rewrite), `mixin/CameraMixin.kt` (FREECAM writes pose), `mixin/FreecamEntityMixin.kt` (redirect to pose), `mixin/EntityRendererMixin.kt` (hand-hide if cheap), `Iustitia.kt` (`tickFreecam` call unchanged — still calls `ReplayState.tickFreecam()`).

## Out of scope (follow-ups)

- **YACL preset UI** — an "Apply preset" / "Create preset" row in the config screen. v1.2.0 is command-only.
- **Preset import/export** — sharing a custom preset file by copy. The JSON is already at `.iustitia/presets/<name>.json`, so manual copy works today; a `/ius exportpreset` command is the polish.

- **Continuous window chunk capture** — capturing every chunk loaded during the whole N-second window, not just at save. Unchanged by v1.2.0.
